## dev 브랜치 전략 (Git Flow)

### 1. 기본 원칙
* **Main/Dev 브랜치 보호:** `dev` 브랜치는 배포 전 단계의 통합 브랜치입니다. 팀장 외에는 직접 `push` 할 수 없습니다.
* **작업 방식:** 모든 작업은 개별 브랜치에서 진행하며, 작업 완료 후 `dev` 브랜치로 **Pull Request(PR)**를 생성합니다.

### 2. 작업 순서 (Workflow)
1.  `dev` 브랜치에서 최신 내용을 `pull` 받습니다. (최신화)
2.  새로운 브랜치를 생성합니다. (네이밍 규칙 준수)
3.  작업 완료 후 커밋합니다.
4.  **중요!:** PR을 생성하기 전, 다시 한 번 `dev` 브랜치의 내용을 내 브랜치로 가져와(`merge` or `rebase`) 충돌을 해결하고 로컬에서 테스트합니다.
5.  Github에서 `dev` 브랜치를 향해 PR을 생성합니다.
6.  팀장의 코드리뷰 및 승인 후 Merge 됩니다.

### 3. 브랜치 명명 규칙 (Naming Convention)
-> notion 내 문서란 참고

### 4. PR 작성 가이드
PR 본문에는 다음 내용이 포함되어야 합니다:
* 작업한 내용 요약
* 빌드 테스트 (PR 생성 전 로컬에서 빌드(서버 실행)가 정상적으로 되는지 확인) 유무
* 최신화 유무 

* (UI 변경 시) 실행 스크린샷 첨부 필수
